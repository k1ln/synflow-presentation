The technology stack

Four layers — each chosen because it does one thing well and stays out of the others' way.

[W3C]
Web Audio API
Real-time audio graph — browser engine
  • Nodes connected into a directed acyclic graph
  • C++ render thread: OscillatorNode, BiquadFilterNode, GainNode, ConvolverNode…
  • AudioWorklet: custom DSP in a high-priority JS thread
browser native · zero install · SIMD-accelerated
[RF]
React Flow (XYFlow)
Visual node graph — UI layer
  • Every Synflow module is a React Flow node with typed ports
  • Edges = signal wires; drag-to-connect fires graph events
  • Custom node components: knobs, waveform previews, ADSR editors
React 18 · XYFlow v12
[EB]
Custom EventBus
Cross-node messaging -- control plane
  • Typed publish / subscribe across any two nodes
  • Independant from React or Audio Engine
  • MIDI, clock ticks, gate signals, CC values -- all routed as events
  • Extremely easy implementation
  • No Redux, no Zustand -- intentionally minimal
zero deps · synchronous · inspectable at runtime
[AG]
Audio Graph Manager
Glue engine -- keeps RF and Web Audio in sync
  • Translates RF edge add/remove into AudioNode.connect / disconnect
  • Owns the AudioContext lifecycle (suspend, resume, sample rate)
  • Node registry: maps each RF node type to its Web Audio factory
  • Hot-swap parameters without tearing down the graph
singleton · runs on main thread · drives both planes
This is the architectural core slide. The key insight: React Flow owns the *visual* graph, Web Audio owns the *audio* graph, and the Audio Graph Manager keeps them in sync. The EventBus is the third bus — it carries non-audio signals (MIDI, clock, gate) that don't belong in either graph. These four pieces compose cleanly because each has a single clear job.

Web Audio API: Two-tier performance model

Tier 1 · Browser engine (C++)
Built-in Audio Nodes
  • OscillatorNode · BiquadFilterNode
  • GainNode · DelayNode · ConvolverNode
  • DynamicsCompressorNode · AnalyserNode
Compiled · SIMD-accelerated · near-zero overhead
Tier 2 · V8 JIT (JavaScript)
AudioWorklet Processor
  • Custom DSP code you write in JavaScript
  • Runs inside process(inputs, outputs)
  • Synflow: Noise, AudioWorklet Oscillator…
Dedicated thread · JIT compiled · no GC inside process()
[t:] Render quantum: 128 samples @ 44 100 Hz = 2.902 ms hard budget per block -- miss it and you get an audible glitch.
Synflow uses BOTH tiers: Oscillator, Filter, Gain hit the C++ tier for free. Only truly custom DSP (noise types, worklet oscillators) drops to JS. This is a deliberate architectural choice.

AudioWorklet JS vs native C++: the gap

Property Native C++ (VST / AU plugin) AudioWorklet (JavaScript)
Execution Compiled machine code V8 JIT — near-native for hot loops
SIMD SSE4 / AVX2 — 4–8 floats per instruction WASM SIMD possible; plain JS: scalar only
Memory / GC Manual heap, zero GC pauses GC won't fire inside process() if you avoid allocations
Thread priority OS real-time audio thread High-priority worklet thread (not hard real-time)
Overhead vs C++ 1× (baseline) ~10–40× slower for math-heavy DSP
Polyphony (sine osc) 1 000+ voices easily ~50–150 voices on modern hardware
Install / deploy Platform binary, DAW plugin format Zero install — runs in any browser tab
The 10-40x gap sounds scary but only matters for very dense polyphony. For prototyping, live performance and most use-cases it's irrelevant — and Synflow's built-in nodes use the free C++ tier anyway.

Multithreading: browser vs native DAW

Web Audio API (Chrome / Firefox)
Main thread · 1 core React UI · XYFlow graph · param updates · EventBus
↓ AudioContext bridge
Audio rendering thread · C++ · 1 core OscillatorNode · BiquadFilter · Gain · Delay · Convolver…
↓ nested
AudioWorklet thread · JS · 1 core (shared) All custom processors compete for this single thread
/!\ No cross-worklet parallelism -- more AudioWorklet nodes all share one thread.
Native DAW (Ableton · Logic · Pro Tools)
Core 0Synth chain A · Bus FX
Core 1Synth chain B · EQ
Core 2Drum bus · Compressor
Core NMaster bus · Limiter
Task graph scheduler assigns plugin chains to cores dynamically each render cycle.
[+] True multi-core parallelism. Each plugin is typically single-threaded internally -- but many chains run simultaneously across all cores.
This is the fundamental architectural difference. Native DAWs scale with CPU core count. Web Audio is pinned to one audio thread — you cannot parallelize across cores. This is a spec limitation, not a Synflow limitation.
